## 1. 项目定位

这是一个用于系统提升全栈、微服务与工程化能力的个人作品项目。以自营实物电商为业务载体，先完成可运行的交易闭环，再逐步引入服务拆分、消息一致性、可观测性和压测，最终形成可演示、可部署、可讲解架构取舍的作品集。

### 个人成长目标

- 掌握 Laravel、Hyperf、Go 在同一业务中的适用边界与协作方式。
- 实践订单、支付、库存等典型复杂业务的状态机与最终一致性。
- 搭建从开发、测试、容器化到监控告警的完整工程链路。
- 沉淀架构图、接口文档、压测报告和故障复盘，支持面试讲解。

### 项目成功标准

**功能完成度**

- 消费者与后台两端可演示完整购买、履约和退款流程。
- 提供模拟支付、模拟物流，不依赖真实商业资质。
- 库存、订单、支付状态可追踪、可重放、可补偿。

**工程完成度**

- 核心逻辑有自动化测试，关键 API 有契约文档。
- 可通过 Docker Compose 一键启动本地完整环境。
- 可从订单号关联日志、链路与业务指标。
- 至少完成一次压测、故障注入和优化复盘。

## 6. 核心交易流程

### 下单与支付（同步响应 + 异步收敛）

1. 结算页调用试算：校验商品、SKU、活动、地址、运费与优惠。
2. 提交订单携带幂等键；服务端再次计价并创建待支付订单。
3. 库存服务原子预占库存，记录库存流水与过期时间。
4. 支付服务创建支付单，返回支付参数。
5. 支付回调验签并幂等更新支付单，发送“支付成功”事件。
6. 订单消费事件转为待发货；超时未支付则关闭订单并释放库存。

### 售后与退款（状态机驱动）

1. 用户按订单项申请售后，系统校验期限、数量与金额。
2. 客服审核；退货退款需生成退货地址和退货单。
3. 仓库确认收货后触发退款；仅退款可审核后直接触发。
4. 支付服务发起原路退款，异步接收退款结果。
5. 成功后更新售后、退款、订单可退金额，并回补库存（按规则）。
6. 失败进入可重试任务和人工处理队列，不回滚已确认事实。

## 7. 状态与业务规则

| 对象   | 主状态流转                                                      | 关键规则                                           |
| ------ | --------------------------------------------------------------- | -------------------------------------------------- |
| 订单   | 待支付 → 待发货 → 已发货 → 已完成；任意合法节点 → 已取消/售后中 | 状态只能按允许路径迁移；每次迁移记录来源和操作日志 |
| 支付单 | 待支付 → 支付中 → 成功/失败/关闭                                | 订单与支付单一对多，任一成功后其余支付单关闭       |
| 库存   | 可售 → 预占 → 扣减；预占 → 释放                                 | Redis 做高并发校验时，MySQL 台账仍是最终事实来源   |
| 售后单 | 待审核 → 待退货/待退款 → 退款中 → 完成/关闭                     | 按订单项累计可退数量、金额，禁止重复超退           |
| 优惠券 | 未领取 → 可用 → 锁定 → 已使用；锁定 → 可用                      | 下单锁定，支付确认使用，订单关闭释放；全程记录流水 |

### 计价规则

- 价格计算统一在服务端完成，前端只负责展示。
- 建议顺序：商品活动价 → 订单级满减 → 优惠券 → 运费。
- 订单级优惠按订单项金额比例分摊，尾差归最后一个有效订单项。
- 退款按已分摊实付金额计算，不超过原支付金额和剩余可退金额。

### 一致性与幂等规则

- 提交订单、支付回调、发货、退款均必须支持幂等。
- 跨服务不使用分布式强事务，采用本地事务 + Outbox/MQ + 补偿。
- MQ 消费至少一次，消费者基于业务唯一键去重。
- 所有自动补偿必须可观测，并提供后台人工重试入口。

## 8. 渐进式技术分工

| 组件                              | 实践职责                                           | 重点学习内容                                 |
| --------------------------------- | -------------------------------------------------- | -------------------------------------------- |
| Vue 3 + Vant + TypeScript + Pinia | 移动 H5、组件封装、请求层、业务状态与埋点          | 组合式 API、类型建模、状态边界、移动端性能   |
| Laravel                           | 第一阶段模块化单体、运营后台 API、RBAC、数据库迁移 | 领域分层、ORM、队列、测试与快速交付          |
| Hyperf                            | 拆出的商品查询、购物车、订单 API                   | 协程、连接池、中间件、服务治理与性能对比     |
| Go                                | 拆出的库存服务与模拟支付服务                       | 并发控制、原子库存、幂等、错误处理与性能剖析 |
| MySQL                             | 权威业务数据与 Outbox 表                           | 索引、事务隔离、锁、慢查询、备份恢复         |
| Redis                             | 缓存、验证码、购物车、库存热点与限流               | 缓存一致性、穿透/击穿、Lua 与分布式锁边界    |
| RabbitMQ                          | 订单、支付、库存领域事件与死信重试                 | 至少一次投递、消费幂等、顺序与消息积压处理   |
| Nacos                             | 拆分后服务注册、环境配置、动态开关                 | 配置版本、灰度变更、多语言客户端接入         |
| ELK                               | 结构化日志、业务审计与错误检索                     | 统一字段规范、索引生命周期与检索面板         |
| SkyWalking                        | 跨服务链路与拓扑                                   | Trace 上下文传播、慢链路定位与采样           |
| Prometheus + Alertmanager         | 技术指标、业务指标、告警规则                       | RED/USE 指标、SLO、告警降噪与容量判断        |

**架构原则**：可以有计划地覆盖全部技术栈，但仍需展示“为什么这样拆”。先用模块化单体跑通业务，再选择库存与支付等边界清晰的领域进行拆分，并保留拆分前后的压测与复杂度对比。

## 16. 个人项目风险与对策

### 主要风险

- 同时学习过多技术，容易长期停留在搭环境阶段。
- 过早拆微服务会放大调试、部署和数据一致性成本。
- 功能不断扩张会导致核心交易链路无法完成。
- 只有代码没有文档与数据，难以证明实际掌握程度。

### 执行对策

- 每阶段必须有可运行 Demo，再引入下一项基础设施。
- 维护需求看板；同时只做一个 P0 任务，P1 不阻塞主线。
- 每次服务拆分都记录 ADR：问题、选项、决策、代价。
- 每两周输出一次复盘：完成项、问题、指标与下一步。
